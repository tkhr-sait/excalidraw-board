# ADR-0003: Firebase代替案の検討結果

## ステータス
承認済み

## コンテキスト
Excalidraw公式ではFirebaseを使用してデータ永続化と画像ファイル管理を行っているが、本プロジェクトではローカルネットワーク完結を要件とするため、Firebase代替案を検討した。

## 現在のFirebase使用箇所

### 1. Firestore（データベース）
```typescript
// シーンデータの保存
{
  sceneVersion: number,
  iv: Bytes,  // 暗号化のための初期化ベクトル
  ciphertext: Bytes  // 暗号化されたシーンデータ
}
```

**使用目的**:
- ルームデータの永続化
- 新規参加者への初期状態提供
- 接続断絶時のフォールバック

### 2. Firebase Storage
```typescript
// ファイルストレージのパス構造
/files/rooms/{roomId}/
```

**使用目的**:
- 画像ファイルの保存（最大3MB）
- ファイルの暗号化保存
- 複数ユーザー間でのファイル共有

### 3. 使用されるFirebase機能
- **Authentication**: ユーザー認証（簡易）
- **Firestore**: ドキュメントデータベース
- **Storage**: バイナリファイル保存
- **トランザクション**: アトミックな更新

## 代替案の検討

### 案1: excalidraw-roomのみ使用（最小構成）
**概要**: Firebase機能を完全に除外し、excalidraw-roomのメモリ管理のみを使用

**メリット**:
- 最もシンプルな構成
- 外部依存なし
- セットアップが簡単

**デメリット**:
- サーバー再起動時にデータ消失
- 大きなファイルを扱えない
- 新規参加者への初期状態提供が困難

**評価**: ❌ 実用性に欠ける

### 案2: ローカルストレージ + サーバーサイドファイル保存
**概要**: ブラウザのlocalStorageとサーバーのファイルシステムを組み合わせ

**メリット**:
- 軽量な実装
- ブラウザでの一時保存
- サーバーでのファイル管理

**デメリット**:
- ブラウザ間でのデータ共有が困難
- localStorageの容量制限
- データ同期の複雑性

**評価**: ⚠️ 限定的な用途

### 案3: SQLite + ローカルファイルシステム
**概要**: サーバーサイドでSQLiteとファイルシステムを使用

**メリット**:
- 軽量なデータベース
- ファイル保存が可能
- トランザクション対応
- バックアップが簡単

**デメリット**:
- 追加実装が必要
- 同時アクセス制御

**評価**: ✅ 実用的な選択肢

### 案4: Redis + ローカルファイルシステム
**概要**: インメモリデータベースRedisとファイルシステムの組み合わせ

**メリット**:
- 高速なデータアクセス
- PubSub機能でリアルタイム通信強化
- データ永続化オプション

**デメリット**:
- 追加のインフラ
- メモリ使用量
- 設定の複雑さ

**評価**: ⚠️ 過剰な構成

### 案5: JSONファイル + ローカルファイルシステム（推奨）
**概要**: シンプルなJSONファイルによるデータ管理

**メリット**:
- 実装が最も簡単
- デバッグしやすい
- 外部依存なし
- 可読性が高い

**デメリット**:
- 同時書き込み制御が必要
- 大量データでのパフォーマンス

**評価**: ✅ プロジェクト要件に最適

## 採用する代替案

### 選定結果: 案5（JSONファイル + ローカルファイルシステム）

#### 実装方針
```
excalidraw-board/
├── data/
│   ├── rooms/
│   │   └── {roomId}.json     # ルームデータ
│   └── files/
│       └── {roomId}/
│           └── {fileId}      # ファイルデータ
```

#### データ構造
```typescript
// ルームデータ（{roomId}.json）
interface RoomData {
  roomId: string;
  version: number;
  lastUpdated: string;
  elements: ExcalidrawElement[];
  appState: AppState;
  files: {
    [fileId: string]: {
      id: string;
      created: number;
      dataURL?: string;
    };
  };
}
```

### 実装する機能

#### 1. データ永続化
- **ルームデータ**: JSONファイルで保存
- **増分保存**: 変更時のみファイル更新
- **バージョン管理**: データ競合の検出

#### 2. ファイル管理
- **画像保存**: ローカルファイルシステム
- **ファイル制限**: 最大サイズとファイル数の制限
- **クリーンアップ**: 未使用ファイルの定期削除

#### 3. 同期処理
- **排他制御**: ファイルロックによる同時書き込み防止
- **エラーハンドリング**: ファイルアクセスエラーの処理
- **フォールバック**: メモリ上のデータで継続

### excalidraw-roomの拡張

#### 必要な機能追加
1. **ファイルAPI**: 画像アップロード・ダウンロード
2. **データ永続化**: ルームデータの保存・読込
3. **クリーンアップ**: 古いデータの削除

#### 実装方法
- **カスタムDockerfile**: 公式イメージをベースに機能追加
- **ボリュームマウント**: データディレクトリの永続化
- **環境変数**: データディレクトリとファイル制限の設定

## 決定事項

### 採用する技術
1. **データ保存**: JSONファイル
2. **ファイル保存**: ローカルファイルシステム
3. **同期**: ファイルロック + バージョン管理

### 実装する機能
1. **ルームデータ永続化**: 自動保存・読込
2. **ファイル管理**: 画像アップロード・ダウンロード
3. **データクリーンアップ**: 古いデータの自動削除

### 制限事項
1. **ファイルサイズ**: 最大3MB（Firebase準拠）
2. **ルーム数**: 制限なし（ディスク容量依存）
3. **同時接続**: excalidraw-room仕様に依存

## 実装スケジュール

1. **フェーズ1**: 基本的なJSON保存機能
2. **フェーズ2**: ファイルアップロード機能
3. **フェーズ3**: クリーンアップとエラーハンドリング

## リスク管理

### 想定されるリスク
1. **データ競合**: 同時書き込みによるデータ破損
2. **ディスク容量**: ファイル蓄積によるディスク圧迫
3. **パフォーマンス**: 大量データでの読込速度

### 対策
1. **排他制御**: ファイルロックの実装
2. **容量監視**: ディスク使用量の定期チェック
3. **最適化**: データ圧縮と不要データ削除
